# novelWebTool — AI 小说写作助手

一个运行在浏览器中的 **AI 辅助小说创作工具**，纯前端架构，数据存储在本地 IndexedDB，通过调用大模型 API 实现从故事构思到正文写作的完整创作流程。

---

## 设计理念

本工具围绕"**AI 辅助而非替代**"的核心原则设计。AI 的能力被约束为两个层面：

1. **查询与建议**：AI 可以主动读取用户已创建的所有素材（人物、世界观、时间线、大纲、章节），在理解全貌后给出创作建议
2. **结构化写入**：AI 可以直接创建和修改素材，但所有操作都沉淀为结构化数据（而非仅停留在聊天文本），用户始终拥有对数据的掌控权

这意味着用户与 AI 的关系是**协作式**的：AI 推进"工程化"工作（搭建框架、补充细节、保持一致性），用户把控"创造性"决策（故事走向、风格调性、关键情节）。

---

**上下文素材**（`buildPrompt`），按用户配置选择性附上：

- 小说基本信息（标题、类型、简介）
- 当前章节标题
- 故事总大纲 / 当前章节简述
- 世界观设定条目
- 人物设定（含人物关系，支持单向↔双向可视化）
- 多轨道时间线（按轨道分组、按顺序排列，含重点标记和跨段事件）
- 前文概要（最近 N 章末尾内容，帮助 AI 理解最近的文风）
- 前文章节结尾参考

### 附件系统：用户主导上下文

用户通过附件栏（AttachmentBar）自行决定哪些素材要附上下文中。AI 虽然有 `get_characters`、`get_world_settings` 等查询工具可以主动查找，但附件系统的设计思想是 **"用户认为重要的，就应该被看到"**——避免 AI 因"觉得没必要查"而遗漏关键信息。

| 附件类别 | 支持粒度筛选 | 说明 |
|---------|------------|------|
| 人物设定 | ✅ 按人筛选 | 创作某些章节时只需关注特定角色 |
| 世界观设定 | ✅ 按条目筛选 | 不同场景只需相关世界观规则 |
| 故事大纲 | ❌ | 全局把控 |
| 本章大纲 | ❌ | 当前章节的规划 |
| 时间线 | ✅ 按轨道筛选 | 关注特定线（如只带感情线） |
| 前文概要 | ❌ | 最近 5 章末尾 500 字 |
| 前文结尾参考 | ❌ | 最近 3 章末尾 300 字 |

人物、世界观、时间线三类支持按条目粒度筛选，因为它们在规模较大时，全量附加会造成 token 浪费和上下文干扰。

附件配置持久化到 localStorage，跨会话保留。

## 关键架构设计

### 人物系统与关系图谱

人物管理不只是一张属性表。每个角色包含姓名、性别、年龄、性格、外貌、背景、关系等字段，更重要的是人物之间可以建立**关系连线**：

- 支持**单向关系**（A → B：暗恋、追杀）和**双向关系**（A ↔ B：师徒、宿敌）
- 关系图可视化呈现，节点为人物，连线为关系（带箭头方向和标签）
- AI 可以通过 `create_relation` / `update_relation` / `delete_relation` 工具自主管理人物关系网
- 在构建新故事时，AI 被要求在创建人物后主动建立人物之间的关系

这种设计让故事中的人物不再是孤立的条目，而是形成一个可交互的关系网络，帮助作者理清人物之间的情感纽带和利益冲突。

### 多轨道时间线矩阵

时间线是本工具的核心创新之一，采用**矩阵式可视化**设计：

- 每一条 **轨道** 代表一条独立的故事线（如"主线""感情线""反派行动线""暗线"）
- **行 = 章节序号**，**列 = 轨道**，形成二维矩阵
- 事件出现在对应轨道的对应章节位置
- 事件支持 **跨段持续**（设置 endOrder），在多个章节中持续显示
- 支持重点标记（⭐ 金色高亮）

这种设计让作者能够直观地管理多线叙事：明线和暗线在同一时间点的交错、不同角色在同一事件中的不同视角、多条故事线的并行推进，都可以在矩阵中一目了然。

### Novel Memory：跨会话持久记忆

标准聊天 AI 在清空对话记录后会"失忆"，这在长篇小说创作中是一个致命问题。Novel Memory 机制解决这个问题：

1. AI 在阅读全部素材后，生成一份结构化的小说记忆摘要
2. 摘要持久化存储（独立于聊天记录），包含：基本设定概览、剧情进展、写作风格、最新发展、待解决问题
3. 每次对话时自动加载到上下文中
4. 用户也可以在素材发生重大变化时手动触发"刷新记忆"

这样即使清空对话、切换模型、重新打开页面，AI 仍然能理解小说的全貌。

### 工具分类与调用状态机

AI 通过 Function Calling 与系统交互，约 40+ 个工具被分为三类：

| 分类 | 行为 | 示例 |
|------|------|------|
| **查询工具** | 自动执行，结果直接喂回 AI | get_characters, get_timeline, get_chapter_content |
| **写入工具** | 自动执行，执行后刷新 UI | create_character, write_chapter_content, update_story_outline |
| **交互工具** | 触发前端 UI 组件，等待用户回应 | ask_questions（弹出问答卡片）, suggest_solutions（弹出方案卡片） |

核心循环（Chat State Machine）：

```
idle → sending → streaming → tool_calling → executing_tools
→ (back to sending | waiting_user | waiting_confirmation)
```

- `waiting_user`：AI 通过交互工具向用户提问或提供多个方案，用户回复后继续
- `waiting_confirmation`：AI 尝试执行破坏性操作时需用户确认

### 数据可移植性

工具内置完整的**导入/导出功能**，支持将整部小说（含所有章节、人物、世界观、时间线、大纲、记忆）导出为 JSON 文件，用户可以自行备份、迁移或用版本管理工具追踪。

## 技术栈

| 技术 | 用途 |
|------|------|
| **Vue 3** | 前端框架 |
| **TypeScript 6** | 类型安全 |
| **Vite 8** | 构建工具 |
| **Pinia** | 状态管理 |
| **Vue Router** | 路由 |
| **Element Plus** | UI 组件库 |
| **IndexedDB** | 本地数据存储 |
| **OpenAI 兼容 API** | AI 对话与工具调用 |

## 项目结构

```
src/
├── components/
│   ├── ai/          # AI 交互组件（问题卡片、方案卡片、写作设置）
│   ├── chapter/     # 章节管理组件
│   ├── character/   # 人物设定 + 人物关系图
│   ├── chat/        # AI 对话面板（输入、气泡、附件栏）
│   ├── common/      # 通用组件（暗色模式切换、卡片渲染等）
│   ├── dialogs/     # 导入/导出弹窗
│   └── world/       # 世界观设定、时间线编辑器、故事大纲
├── composables/     # useChatSend — 上下文构建与消息发送
├── config/          # API 预设列表、系统提示词 SYSTEM_PROMPT
├── stores/          # 5 个 Pinia 仓库（novel、chapter、material、chat、app）
├── types/           # 全部 TypeScript 类型定义
├── utils/
│   ├── ai/          # 工具定义（definitions）、处理器（handlers）、流式请求（stream）、分类（categories）
│   ├── db.ts        # IndexedDB 封装
│   ├── prompt.ts    # PromptBuilder：结构化上下文构建
│   └── id.ts        # UUID + 时间戳生成
└── views/           # HomePage（小说列表）、NovelDetail（工作台）
```

## 快速开始

```bash
# 安装依赖
npm install

# 启动开发服务器
npm run dev

# 构建生产版本
npm run build
```

## 使用说明

1. **配置 API**：点击右上角"API 设置"，填入 API 地址和 Key
2. **新建小说**：设置小说名称、类型和简介
3. **完善设定**：在小说详情页中管理人物、世界观、时间线
4. **AI 写作**：在右侧聊天面板输入需求，AI 会根据现有素材辅助创作
